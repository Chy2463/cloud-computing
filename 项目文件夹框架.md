下面给你一份**推荐的完整项目文件夹框架**，以及**三位组员分别负责哪个文件夹/模块**（保证每个人都能在个人报告里写出至少 2 个具体任务 + 对应 artefacts）。

---

## 1) 项目文件夹框架（推荐）
> 一个 GitHub 仓库，目录清晰对应 6 个组件 + 部署/测试材料。

```
campus-buzz/
  README.md
  docker-compose.yml
  .env.example

  docs/
    architecture.md              # 系统架构图/端到端流程说明
    api.md                       # 接口清单（可选，但很加分）
    test-cases.md                # TC1-TC8 用例表（验证证据）

  presentation-service/
    app/                         # Flask(或前端)代码
    Dockerfile
    requirements.txt
    README.md                    # 如何本地运行/容器运行（可选）

  workflow-service/
    app/
    Dockerfile
    requirements.txt
    README.md

  data-service/
    app/
    migrations/                  # 可选（若你们做简单建表脚本）
    storage/                     # SQLite 文件挂载目录（容器 volume）
    Dockerfile
    requirements.txt
    README.md

  functions/
    submission_event/
      lambda_function.py
      requirements.txt
      README.md
    processing/
      lambda_function.py
      requirements.txt
      README.md
    result_update/
      lambda_function.py
      requirements.txt
      README.md

  scripts/
    deploy_ec2.sh                # 可选：EC2 部署脚本/命令集合（不强制）
    ecr_push.sh                  # 可选：ECR push/pull 步骤
    seed_test_cases.sh           # 可选：一键跑测试用例

  tests/
    cases/
      tc1_incomplete_missing_field.json
      tc2_invalid_date.json
      tc3_short_description.json
      tc4_general_approved.json
      tc5_opportunity.json
      tc6_academic.json
      tc7_social.json
      tc8_precedence.json
    postman_collection.json      # 可选
```

### 关键约定（避免后面合并冲突）
- 所有服务/函数的配置项统一写在 `.env.example`，本地用 `.env`（不提交 secrets）。
- `docker-compose.yml` 放在仓库根目录，负责把 3 个容器拉起来并建立内部网络。
- Lambda 每个函数一个子文件夹，便于分别部署和写个人报告“我负责哪个函数”。

---

## 2) 三位组员负责哪个文件夹（推荐分工）
### 组员 A（Presentation 负责人）
主要负责：
- `presentation-service/`
- `docs/` 里的 UI 相关说明（可选）
- （可选）`tests/` 中与展示相关的说明（如如何在页面触发 TC 用例）

典型可写进个人报告的 artefacts：
- `presentation-service/app/*`
- `presentation-service/Dockerfile`
- 表单字段校验/页面渲染逻辑、调用 Workflow 的 API 代码

---

### 组员 B（Workflow + 集成入口负责人）
主要负责：
- `workflow-service/`
- `docker-compose.yml`（建议由 B 主导，因为它是“集成核心”）
- `docs/api.md`（接口文档，写清楚对外 API，强烈加分）

典型 artefacts：
- `workflow-service/app/*`
- `workflow-service/Dockerfile`
- `docker-compose.yml`（网络、端口、环境变量）
- 调用 Lambda 的代码（invoke SubmissionEvent），以及 `/api/submissions`、`/api/submissions/<id>`、`/api/submissions/<id>/result`

---

### 组员 C（Data + Lambda 规则负责人）
主要负责：
- `data-service/`
- `functions/processing/`（必须是你们规则核心）
- `functions/result_update/`
- （可选）`functions/submission_event/`（如果工作量大，可把这个给 B）

同时建议由 C 主导：
- `docs/test-cases.md`（TC1–TC8 用例表 + 期望输出），因为和规则实现绑定最紧

典型 artefacts：
- `data-service/app/*`、SQLite 建表/模型、CRUD API
- `functions/processing/lambda_function.py`（规则实现）
- `functions/result_update/lambda_function.py`（写回逻辑）
- 用例文件 `tests/cases/*`（验证证据）

---

## 3) “共同负责但指定主负责人”的文件（避免没人管）
- `README.md`：B 主写，A/C 补充各自模块运行说明
- `docs/architecture.md`：B 主写（端到端流程 + 组件交互），C 补充规则说明，A 补充页面流程
- `scripts/`：谁做部署谁维护（通常 B 或 C）
- 演示视频：A 主导录制（因为有 UI），B/C 配合展示 CloudWatch Logs/后端日志
