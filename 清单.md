下面给出一份**可直接作为你们实现与讨论依据**的“最小接口清单（每个服务/函数的输入输出 JSON）+ 里程碑顺序（先跑通再完善）”。技术栈方面：项目文件**没有强制指定语言/框架**；结合你们 Lab（Flask、Docker、Lambda Python3.12），我选用**最容易实现但不至于过于简单**的一套。

---

## 一、技术栈选择（在文件未规定时的推荐）
### 容器服务（3 个 Containers）
- 语言/框架：**Python 3.12 + Flask**
- API 通信：HTTP/JSON
- 数据服务：**SQLite（文件持久化）** + Flask API  
  - 理由：比纯内存更可靠（重启不丢），比上 RDS 简单（省配置/费用），不算“过于简单”。
- 部署：EC2 上 **docker-compose** 跑 3 个容器（符合你们 Lab2 风格）
- 镜像：可选 **ECR**（符合 Lab2）

### Serverless（3 个 Functions）
- 平台：**AWS Lambda**
- 运行时：**Python 3.12**（你们 Lab3 就是这个）
- 日志：**CloudWatch Logs**（用于验证与演示证据）

### 触发方式（你选的方案 1）
- **Workflow Service 异步 invoke Submission Event Lambda**
- Lambda 链：SubmissionEvent → Processing → ResultUpdate

---

## 二、数据与状态约定（接口里会用到）
### 1) 提交输入（Submission Input）
必填字段（按项目文件）：`title, description, location, date, organiser`

### 2) 处理输出（Result）
- `status`: `APPROVED | NEEDS_REVISION | INCOMPLETE`
- `category`: `OPPORTUNITY | ACADEMIC | SOCIAL | GENERAL`
- `priority`: `HIGH | MEDIUM | NORMAL`
- `note`: 简短说明（你们可自定义措辞，但逻辑不能改）

### 3) 记录结构（建议 Data Service 存储）
```json
{
  "submission_id": "uuid-or-shortid",
  "input": { "title":"", "description":"", "location":"", "date":"", "organiser":"" },
  "result": { "status":"...", "category":"...", "priority":"...", "note":"..." },
  "created_at": "ISO-8601",
  "updated_at": "ISO-8601"
}
```
> 初始创建时 result 可先写成：
```json
{"status":"PENDING","category":"GENERAL","priority":"NORMAL","note":"Pending processing"}
```
（注意：最终展示给用户时必须是三种终态之一；PENDING 仅内部过渡用）

## 三、最小接口清单（按组件逐个列出）

# A. Presentation Service（容器）
职责：接收用户表单、展示结果；**不直接存数据**，只调用 Workflow。

### 1) 提交页面（可选，但一般需要）
- `GET /`  
返回 HTML（表单页面），无需 JSON 约定。

### 2) 提交表单（前端 → Presentation）
- `POST /submit`（表单或 JSON 均可，内部转发给 Workflow）
- 若用 JSON，输入：
```json
{ "title":"", "description":"", "location":"", "date":"YYYY-MM-DD", "organiser":"" }
```
- 输出（页面跳转或 JSON）：
```json
{ "submission_id":"...", "message":"submitted" }
```

### 3) 查询展示（用户 → Presentation）
- `GET /status/<submission_id>`
- Presentation 内部调用 Workflow `GET /api/submissions/<id>`，然后渲染页面

> Presentation 也可以提供纯 API 形式（方便测试），但不是必须：
- `GET /api/status/<submission_id>` → 直接透传 Workflow 返回 JSON

---

# B. Workflow Service（容器，系统入口/编排）
职责：创建记录、触发后台处理、提供对外查询、接收结果写回（给 Lambda 用）。

### 1) 创建提交 + 触发处理（Presentation → Workflow）
- `POST /api/submissions`
- 输入 JSON（必填字段由 Processing 最终判定，但这里至少接收这 5 项）：
```json
{
  "title":"...",
  "description":"...",
  "location":"...",
  "date":"2026-04-15",
  "organiser":"..."
}
```
- 行为：
  1) 调 Data Service `POST /records` 创建初始记录，拿到 `submission_id`
  2) **异步 invoke** Submission Event Lambda：传 `submission_id`
- 成功输出：
```json
{ "submission_id":"...", "message":"created_and_processing_triggered" }
```
- 失败输出（示例）：
```json
{ "error":"DATA_SERVICE_UNAVAILABLE" }
```
HTTP 码建议：成功 201；Data 不可用 503。

### 2) 查询提交与结果（Presentation → Workflow）
- `GET /api/submissions/<submission_id>`
- 输出（必须能让 Presentation 展示 status/category/priority/note）：
```json
{
  "submission_id":"...",
  "input": { "title":"...", "description":"...", "location":"...", "date":"...", "organiser":"..." },
  "result": { "status":"APPROVED", "category":"GENERAL", "priority":"NORMAL", "note":"..." }
}
```
- 若不存在：404
```json
{ "error":"NOT_FOUND" }
```

### 3) 写回处理结果（Result Update Lambda → Workflow）
- `POST /api/submissions/<submission_id>/result`
- 输入：
```json
{
  "status":"APPROVED|NEEDS_REVISION|INCOMPLETE",
  "category":"OPPORTUNITY|ACADEMIC|SOCIAL|GENERAL",
  "priority":"HIGH|MEDIUM|NORMAL",
  "note":"..."
}
```
- 行为：转调 Data Service `PATCH /records/<id>/result`
- 输出：
```json
{ "submission_id":"...", "updated": true }
```
- 若 submission 不存在：404 `{ "error":"NOT_FOUND" }`

> 这个设计的关键好处：Lambda **只需访问 Workflow（一个对外 API）**，不需要直连 Data Service（避免 VPC/内网复杂度）。

---

# C. Data Service（容器，存储层）
职责：存取记录。建议只对容器内网开放，不对公网开放。

### 1) 创建记录（Workflow → Data）
- `POST /records`
- 输入：
```json
{ "input": { "title":"", "description":"", "location":"", "date":"", "organiser":"" } }
```
- 输出：
```json
{ "submission_id":"...", "created_at":"...", "result": { "status":"PENDING", "category":"GENERAL", "priority":"NORMAL", "note":"Pending processing" } }
```

### 2) 查询记录（Workflow → Data）
- `GET /records/<submission_id>`
- 输出：同“记录结构”
- 不存在：404

### 3) 更新结果（Workflow → Data）
- `PATCH /records/<submission_id>/result`
- 输入：
```json
{ "result": { "status":"...", "category":"...", "priority":"...", "note":"..." } }
```
- 输出：
```json
{ "submission_id":"...", "updated": true, "updated_at":"..." }
```

---

# D. Lambda Functions（3 个，事件链）
下面的 payload 设计保证链路简单、可追踪（CloudWatch 打印 submission_id）。

## 1) Submission Event Function（Lambda）
- 触发方式：Workflow 异步 invoke
- 输入 event（Workflow 传）：
```json
{ "submission_id":"..." }
```
- 输出（给下一步；你们也可以直接在本函数内 invoke 下一步，不依赖返回值）：
```json
{ "submission_id":"...", "next":"processing" }
```
- 行为：invoke Processing Function（建议同步，便于拿到结果再传下去）

## 2) Processing Function（Lambda，规则核心）
- 输入 event：
```json
{ "submission_id":"..." }
```
- 行为：
  1) 调 Workflow：`GET /api/submissions/<id>` 获取 input
  2) 严格按项目 PDF 规则计算：status/category/priority/note
- 输出（给 Result Update Function）：
```json
{
  "submission_id":"...",
  "result": {
    "status":"APPROVED|NEEDS_REVISION|INCOMPLETE",
    "category":"OPPORTUNITY|ACADEMIC|SOCIAL|GENERAL",
    "priority":"HIGH|MEDIUM|NORMAL",
    "note":"..."
  }
}
```

## 3) Result Update Function（Lambda）
- 输入 event：Processing 输出
- 行为：调用 Workflow
  - `POST /api/submissions/<id>/result` 写回
- 输出：
```json
{ "submission_id":"...", "updated": true }
```

## 四、里程碑顺序（先跑通 → 再完善）
每一步都对应“可验收成果”，避免最后集成爆炸。

### Milestone 0（0.5 天）：对齐协议与仓库
- 定下：上述 JSON 字段名、endpoint 路径、端口、submission_id 格式
- 建 GitHub 仓库结构（presentation/workflow/data/functions）
- 写 README 的骨架（先留占位符）

验收：能在本地起空服务，接口路径不再改来改去。

### Milestone 1（1 天）：Data Service + Workflow 基础打通（不接 Lambda）
- Data：实现 `POST/GET/PATCH`
- Workflow：实现 `POST /api/submissions`（先只创建记录，不触发 Lambda）
- Workflow：实现 `GET /api/submissions/<id>`

验收：curl 创建记录 → curl 查询能看到 PENDING result。

### Milestone 2（1 天）：Presentation → Workflow 端到端（仍不接 Lambda）
- Presentation 表单提交到 Workflow
- 展示查询页面（看到 PENDING 也可以）

验收：浏览器可操作（提交后拿到 submission_id，能查到记录）。

### Milestone 3（1–2 天）：3 个 Lambda 链路跑通（先写“固定结果”）
- 三个 Lambda 都能被调用，CloudWatch 有日志
- Workflow 能异步 invoke SubmissionEvent
- ResultUpdate 能调用 Workflow 写回固定 result（比如 GENERAL/NORMAL/APPROVED）

验收：提交后等待几秒，查询显示 result 被更新（哪怕是固定值）。

### Milestone 4（1–2 天）：Processing 规则实现 + 测试用例覆盖
- Processing Function 严格实现：
  - 必填缺失 → INCOMPLETE（最高优先）
  - 日期格式不对 → NEEDS_REVISION
  - 描述 < 40 → NEEDS_REVISION
  - 分类关键词与优先级（OPPORTUNITY > ACADEMIC > SOCIAL > GENERAL）
  - priority 映射
- 跑至少 6–8 组用例（你们报告也用这些做验证证据）

验收：每个用例输出与规则一致，CloudWatch + 页面展示一致。

### Milestone 5（1 天）：AWS 部署固化 + 视频/报告素材
- EC2 上 docker-compose 部署（可选 ECR）
- 安全组只放行 Presentation 端口
- CloudWatch logs 截取调用链关键日志
- 录视频：端到端演示 2–3 个用例 + logs

验收：组内任何人按 README 能复现部署/演示流程。
